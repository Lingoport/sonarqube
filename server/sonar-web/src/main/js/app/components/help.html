Navigating the Lingoport Dashboard

Dashboard Projects

When you first access the Lingoport Dashboard from your browser, you will see the Projects list.

These projects are created using the Lingoport Dashboard Client installer. The Lingoport Dashboard Client installer creates scripts (DashboardUpload, GlobalyzerImportScanReport, GlobalyzerScanReport, LRMScan) that you can call from your continuous build environment or from a user's system to run your Globalyzer Project Scans and/or your Lingoport Resource Manager reports, and then upload the results to an associated Lingoport Dashboard Project (with the same name as your Globalyzer Project and/or your LRM Project). The javademo Dashboard project shows both the Globalyzer results for the internationalization side and the LRM results for a status of the localized resource files for the software project.

To add your own dashboard projects, run the Lingoport Dashboard Client installer. Then run the customized scripts to generate the reports and to upload the data to the Lingoport Dashboard.


The following lists the information you'll see when on the Projects page:


  • Quality Gate (QG): The project status with respect to its quality gate.
  • NAME: The name of the Lingoport Dashboard project.
  • VERSION: The version of the Lingoport Dashboard project, as set in the sonar-project.properties file, which is created by the DashboardUpload script. The default is 1.0.
  • Lines of Code (LOC): The total lines of code from all the Globalyzer Scan data.
  • TECHNICAL DEBT: Total effort (in days) to fix all the issues on the component and therefore to comply to all the requirements
  • LAST ANALYSIS: The last date the Dashboard was updated.

Note: To return to the Projects page, click on Dashboards -> Home in the menu bar.


A Single Project's Dashboard

Click on a project name in the Projects page to see the project overview. For example, click on the javademo project. The following sections describe the various widgets of the main Dashboard page.


Time changes (drop down)

See the project data over various time spans to see how i18n and L10n issue counts have gone up or down.


Globalyzer Summary

This widget totals all the Globalyzer Scan results for your Globalyzer project in the 4 categories: Embedded Strings, Locale-Sensitive Methods, General Patterns, and Static File References.

Click on each number to drill down on the specific issues identified by Globalyzer.


Resource Manager Summary Report

This widget provides a quick overview of the current translation status of the resource files based on the LRM project configuration.

  • Base Resource Files: the number of resource files in the default locale expected to be translated to the target locales.
  • Base Resource Words: the number of words found in the resource files and expected to be translated to the target locales.
  • Target Locales: the total number of target locales for this project. For instance, if you are targeting fr_FR, fr_CA, and de_DE, it would be three target locales.
  • % Complete: the percentage of translated words over all locales. For instance, if all the files are completed translated to fr_FR and fr_CA, and none to de_DE, the % Complete would be 66%.
  • Last Prepped Kit Version: the kit number of the last kit sent for translation.
  • Last Prepped Kit Date: when the L10n kit was last sent for translation, in UTC.

Issues and Technical Debt

This widget shows the issues and technical debt identified by Globalyzer and/or by LRM as a combined set of issues.

Click on each number to drill down on the specific issues identified by Globalyzer or LRM.


Issues Time Line

This widget shows the overall progress of the project using the total number of issues and the total number of remaining words to be translated over time.


Globalyzer & i18n

If you have configured your Dashboard project to show Globalyzer data, click on the Globalyzer entry on the More drop-down menu to see more internationalization details.


Globalyzer Summary

This widget totals all the Globalyzer Scan results for your Globalyzer project in the 4 categories: Embedded Strings, Locale-Sensitive Methods, General Patterns, and Static File References.

Click on each number to drill down on the specific issues identified by Globalyzer.


Globalyzer Scans

This widget gives you individual Globalyzer Scan information, showing you the associated Rule Set, the total number of i18n issues, and then the number of lines of code and number of source code files that were processed during the scan.


Globalyzer Issues

The Issues widget is where you can see your i18n issues broken down by severity. We've mapped Globalyzer priority 1 issues to Critical; priority 2 issues to Major; and priority 3 and 4 and 5 issues to Minor. You can login as Admin and modify this mapping by selecting Settings in the menu bar and then Configuration and Globalyzer Violations Widget Plugin.


As mentioned on the Projects page, the Rules Compliance is a measure of how far you are along in your internationalization effort.

Click on the total issues to drilldown to more specific details.


Issues Time Line

As you update your Dashboard's Projects with your latest Scan data, you'll see the changes reflected in the time line, indicating how your internationalization efforts are proceeding.


Resource Manager & L10n

If you have configured your Dashboard project to show Lingoport Resource Manager data, click on the Resource Manager entry on the More drop-down menu to see more resource translation details.


Source Issues

These issues are found in the source directory, for instance after checking out a project on a build machine or a developer system. Click on the number of issues to drill down to the exact file and issues detected by LRM.


Translation Issues

These issues are found when trying to import a target locale translation; for example, after a translation group has sent back the French files for integration. The issues will be listed, even if the files themselves will not be part of the source directories. It helps the translation group see what LRM has identified as an issue and points to the specific location in the file where the issue was found so it is easier to correct.


Next Prep Kit Content

This widget provides information on the contents of the next prep kit.
For each target locale, this widget shows the number of files, number of keys and number of words that will be sent out in the next prep kit. If a target locale is missing from the widget, then the next prep kit will not contain any files for that target locale.

If there are any critical errors, such as a duplicate key, then this widget will display the number of critical errors.


Prep Kit Due Dates

This widget provides a target locale by target locale report of the status of all outstanding prep kits. For each target locale, this widget shows:

  • Kit #: the prep kit version number
  • Locale: The locale of the prep kit.
  • # of Files: the number of files in the prep kit
  • Sent Date: the date that the prep kit was created
  • Due Date: the date the prep kit is due. Due date may be blank if no due date was set via LRM
  • Days Late: the difference, in days, between the sent date and the due date.
    • The cell is yellow if the prep kit is 7-13 days late.
    • The cell is red if the prep kit is at least 14 days late

If there are any critical errors, such as a duplicate key, then this widget will display the number of critical errors.


Resource Manager Productivity Report

This widget provides a target locale by target locale report of how fast the overall process takes.

  • Base Resource Files: the number of resource files in the default locale expected to be translated to the target locales.
  • Base Resource Keys: the total number of keys in the resource files with text to be translated.
  • Base Resource Words: the total number of words in the resource files expected to be translated in each target locale.
  • Last Sent Kit: the kit number of the latest kit sent for translation.
For each target locale, this widget shows how many words per day in the overall process (from the time the kit is sent to the time the translation is imported correctly into the source directories). It then gives an estimate based on the historical productivity data and the number of remaining words to translate of how long it may take for that target locale.


Resource Manager Completion Report

This widget provides a target locale by target locale report of how complete the translation is.

  • Base Resource Files: the number of resource files in the default locale expected to be translated to the target locales.
  • Base Resource Keys: the total number of keys in the resource files with text to be translated.
  • Base Resource Words: the total number of words in the resource files expected to be translated in each target locale.
  • Last Sent Kit: the kit number of the latest kit sent for translation.
For each target locale, this widget shows the percentage of completion (number of base words translated/number of base words to translate), how many files still need to be translated, how many keys need translation, and how many words need to be translated.

This provides a clear status of which locales are ready for deployment.


Time Line

This time line shows the number of remaining words to be translated, the total remaining files and the translation issues over time.

Dashboard Checks

The following list describes briefly each one of the checks performed by the Dashboard. There are two categories of checks: Globalyzer checks and LRM checks.


Globalyzer Checks

  • Globalyzer Embedded String Check: checks for strings in the code which are candidates for eternalization. The priority of the check depends on how likely the string needs to be externalized.
  • Globalyzer Locale-Sensitive Method Check: checks for methods calls which may lead to incorrect behavior depending on the locale or the encoding, such as formatting a date. The priority of the check depends on how likely the method needs to be refactored for a global market.
  • Globalyzer General Pattern Check: checks for patterns in the code which may lead to incorrect behavior depending on the locale or the encoding, such as an ASCII file encoding. The priority of the check depends on how likely the code using the pattern needs to be refactored for i18n.
  • Globalyzer Static File Reference Check: checks for files such as a video file or an HTML licensing agreement, which may need to be handled differently for different countries. The priority of the check depends on how likely the code referencing the file needs to be refactored.

LRM Checks

LRM source base checks apply to the resource files to be translated to other languages; for instance, resources.properties or resources.resx, but not resources_fr_FR.properties or resources.fr-FR.resx for applications initially written in U.S. English.

  • LRM Source Base Date Format Check: The base text associated with this key may be a date format
  • LRM Source Base Duplicate Key Check: checks that each key appears only once in each resource file. If a single key appears multiple times, the code will not know which key to use and the file should not be sent for translation.
  • LRM Source Base Encoding Check: checks that the encoding of the resource file matches the encoding specified in the LRM project configuration. If the LRM project states that the resource file should be in ASCII and it is in UTF-16LE, it should not be sent for translation and it may not be read correctly by the application.
  • LRM Source Base File Too Big Check: the resource file exceeds the maximum size supported by the Lingoport Resource Manager. It may indicate that the file is not an actual resource file; it could be a video file for instance.
  • LRM Source Base HTML Check: There is HTML in the text associated with a key in the base resource.
  • LRM Source Base Modified Text Check: The base text has been modified and the translation has not been updated
  • LRM Source Base No Key Check: a key/value pair is missing the key portion, making the text irretrievable by the application.
  • LRM Source Base No Text Check: a key/value pair is missing the value portion, making the key irrelevant in the resource file.
  • LRM Source Base Not In LRM Check: the resource file is not part of the LRM project.
  • LRM Source Base Unknown File/Line Format Check: Unknown format for file/line or empty file - The base file/line could not be parsed

LRM source translation checks apply to the resource files which have been translated to other languages and are part of the source code; for instance, resources_fr_FR.properties and resources.fr-FR.resx but not resources.properties or resources.resx for applications initially written in U.S. English with French/France as a target language.

  • LRM Source Translation Date Format Check: The translated text associated with this key may be a date format
  • LRM Source Translation Duplicate Key Check: checks that each key appears only once in each translated file.
  • LRM Source Translation Empty Key in Translated File Check: a key/value pair is missing the key portion in the translated file, making the text irretrievable by the application.
  • LRM Source Translation Encoding Check: checks that the encoding of the translated file matches the encoding specified in the LRM project configuration. If the translated file has the wrong encoding, it may not be read correctly by the application.
  • LRM Source Translation File Too Big Check: the translated resource file exceeds the maximum size supported by the Lingoport Resource Manager. It may indicate that the file is not an actual resource file; it could be a video file for instance.
  • LRM Source Translation HTML Check: There is HTML in the text associated with a key in the translated resource.
  • LRM Source Translation Missing Key Check: The identifier/key is missing in the translated file in source.
  • LRM Source Translation Missed Translation Check: the text in the translated resource file is the same as in the original file.
  • LRM Source Translation Missing File Check: the translated resource file is missing. The LRM project configuration calls for this locale.
  • LRM Source Translation No Text Check: a key/value pair is missing the value portion in the translated file, making the key irrelevant in the translated file.
  • LRM Source Translation Parameter Mismatch Check: This translated resource parameters do not match the base resource parameters
  • LRM Source Translation Retranslate Check: This text needs to be retranslated for the translated resource.
  • LRM Source Translation Unknown File/Line Format Check: Unknown format for file/line or empty file - -The translated resource file/line could not be parsed

LRM translation checks apply to the translated files during the import process. Those files are not part of the source code.

  • LRM Translation Duplicate Key Check: checks that each key appears only once in each translated file to be imported.
  • LRM Translation Empty Key Check: a key/value pair is missing the key portion in the translated file to be imported, making the text irretrievable by the application.
  • LRM Translation Encoding Check: checks that the encoding of the translated file to be imported matches the encoding specified in the LRM project configuration. If the translated file has the wrong encoding, it may not be read correctly by the application.
  • LRM Translation File Already Processed Check: the translated resource file to be imported has already been imported for this kit version.
  • LRM Translation File Too Big Check: the translated resource file to be imported exceeds the maximum size supported by the Lingoport Resource Manager. It may indicate that the file is not an actual resource file; it could be a video file for instance.
  • LRM Translation Missed Translation Check: the text in the translated resource file to be imported is the same as in the original file.
  • LRM Translation Missing File Check: the translated resource file is missing from the kit to import.
  • LRM Translation Missing Key Check: the translated file to be imported is missing a key from the original file.
  • LRM Translation Missing or Invalid Tag Check: each file copied to a prep kit has a tag which identifies its purpose. This tag has been tampered with or is missing, making the import impossible.
  • LRM Translation Parameter Mismatch Check: The translation parameters do not match the prep kit's source file parameters.
  • LRM Translation No Text Check: a key/value pair is missing the value portion in the translated file.
  • LRM Translation Non-Writable File Check: the translated file cannot be copied to the source directory.
  • LRM Translation Source Location Not Found Check: the location where the translated file should be copied does not exist.

Finally, if there was any other error on resource files, the following error is shown:

  • LRM Unknown Error Check: some other resource file error occurred.
